# 7 天让 AI 写完一个项目管理系统，我踩了哪些坑？

# 1 先说结论

我用 Claude Code 花 7 天时间，从零到一写了一个项目管理系统（Plane Lite）。后端 Spring Boot，前端 Vue 3，5 张表，25 个 API，7700+ 行代码。

听上去很顺，但过程完全不是。

**前 3 天几乎一半时间在返工**，原因不是 AI 写不出代码——恰恰相反，它写得太快了，快到你还没搞清楚方向，代码已经堆上去了。

这篇文章不是教程，是复盘。记录的是：

> 1. AI 编程从"能跑"到"能用"，中间差了什么？
> 2. 踩过哪些坑，怎么一步步建起了一套工程化流程？
> 3. 最终形成的工作流长什么样？

---

# 2 起点：让 AI 直接写

Day 1，事情很简单：给 Claude Code 一份需求文档，告诉它"建一个 Spring Boot 项目，先实现用户注册登录"。

它很快就写完了。能跑，能登录，看起来没问题。

于是 Day 2，继续加功能——工作区管理、项目管理。

**然后问题就来了。**

## 2.1 Day 2：三个炸弹

**炸弹一：版本不兼容**

AI 选了 Spring Boot 3.2 + MyBatis-Plus 3.5.5，这两个版本放在一起跑不起来。关键是——我让它修，它修完又给我换回去了。同一个错误，反复犯了三次，浪费了半小时。

**炸弹二：代码和方案对不上**

实体类里有个 `leadId` 字段，但数据库建表脚本里压根没有 `lead_id` 这一列。启动直接报错：`Unknown column 'lead_id'`。

**炸弹三：配置不检查**

JWT secret 用了默认值，才 55 个字符（440 bits），但 HS512 要求至少 512 bits。MySQL 密码也是随手写的默认值。

> **根本原因不是 AI 能力不够，而是没有验证机制。**
>
> - 技术选型不验证兼容性
> - 写完代码不和方案对比
> - 启动失败了才知道配置有问题

这让我意识到一件事：

**AI 编程真正的瓶颈不是"写代码"，而是"不犯错"。**

# 3 第一次转变：建质量门

Day 2 的教训逼出了第一个关键动作——**把质量检查写成文档**。

为什么要写文档？因为 AI 没有持久记忆。你口头告诉它"下次注意检查版本兼容性"，下一个会话它就忘了。**唯一能让 AI "记住"的方式，是把流程写进文件。**

于是建了一个**四层质量保障机制**：

| 层 | 名称 | 干什么 | 什么时候做 |
| --- | --- | --- | --- |
| Layer 1 | Plan Mode | 验技术栈兼容性、写完整 DDL、列配置清单 | 开始写代码之前 |
| Layer 2 | Implementation | 实体和数据库对齐、DTO 加校验、Service 加事务 | 写代码的过程中 |
| Layer 3 | Code Review | 代码和方案对比、实体和数据库对比 | 写完代码之后 |
| Layer 4 | Testing | 跑单元测试、跑 API 测试、启动验证 | 审查通过之后 |

简单来说就是：**写之前先想清楚，写的时候自己查，写完了让别人查，查完了再测。**

流程图长这样：

```
Plan Mode（验证方案）
    ↓
Implementation（编码 + 自检）
    ↓
Code Review（代码审查）
    ↓
Testing（自动化测试）
    ↓
Commit
```

这个机制从 Day 3 开始生效。版本不兼容的问题再也没出现过。

> **关键洞察**：对 AI 来说，流程 > 能力。好的流程能让普通模型产出稳定的代码，没有流程的话，再强的模型也会翻车。

# 4 第二次转变：给 AI 加"护栏"

有了四层机制就万事大吉了吗？并没有。

Day 3-4，新的问题出现了：**AI 会跳过流程。**

## 4.1 "自觉"靠不住

我给 AI 定义了一个叫 `implement-day` 的 Skill（可以理解为一个自动化工作流脚本），告诉它"按这个流程走"。但实际执行的时候：

| 该做什么 | 实际做了什么 |
| --- | --- |
| 通过 implement-day Skill 启动 | 看到任务直接开始写代码，完全跳过 Skill |
| 写完代码先 Code Review | 编译通过就直接开始测试，跳过审查 |
| 测试前先查 API 文档 | 凭记忆猜 API 格式，用 username 登录（正确的是 email） |

结果呢？测试脚本写错了，反复调试，浪费了又一个小时。

**根本原因：靠 AI "自觉"遵守流程，永远不可靠。**

## 4.2 三次迭代加护栏

于是 `implement-day` Skill 经历了三轮迭代：

**V1.1 — 加文档查阅步骤**

测试之前，强制要求先读三个文件：
- API 文档（确认登录接口的准确格式）
- 上一个 Day 的测试报告（复用已验证的逻辑）
- 数据库脚本（确认测试用户的凭据）

**V1.2 — 加"真实响应优先"规则**

禁止凭想象写测试脚本。正确流程是：
1. 先手动调一次 API，拿到真实的 JSON 响应
2. 基于真实响应写提取逻辑
3. 一个接口一个接口地测，不要一口气写完再跑

**V2.0 — 加强制检查点**

每个阶段开头加了一个 3-5 秒的前置检查：

```
阶段 0（开始时）：
  ✅ 是否通过 Skill 进入的？ → 如果不是，立即停止
  ✅ 工作目录对不对？
  ✅ 有没有未完成的前置阶段？

阶段 3（Code Review 前）：
  ✅ 编译通过了吗？

阶段 4（Testing 前）：
  ✅ Code Review 做了吗？ ← 这是最容易被跳过的
```

> **关键洞察**：前置检查只需要 3 秒，但能避免 30 分钟的返工。这是 ROI 最高的投入。

# 5 第三次转变：设计不能边写边改

Day 5，开始写前端。登录页面、注册页面，功能不复杂。

但这一天的效率数据让我反思：

| 项目 | 时间 |
| --- | --- |
| 计划编码时间 | 2 小时 |
| 实际编码时间 | 1 小时 |
| 设计调整时间 | 2+ 小时 |
| 设计迭代次数 | 10+ 次 |

发生了什么？AI 先用了暗色风格，我觉得不对，改成亮色。然后对比度不够，改。输入框样式不好看，改。按钮颜色不对，改。字体不满意，改。**改了十几轮，一半时间花在反复调整视觉效果上。**

根本原因很简单：**编码之前没有确认视觉设计方案。**

于是加了一个规则（ADR-003）：

**在技术方案阶段，先回答 4 个设计问题：**

| 问题 | 答案示例 |
| --- | --- |
| Q1: 色调定位 | 亮色系 |
| Q2: 品牌色 | 主色 #1C1410，强调色 #D4870A，背景 #FAF7F2 |
| Q3: 字体搭配 | 标题 DM Serif Display，正文 JetBrains Mono |
| Q4: 参考设计 | 参照 Linear 的简约风格 |

确认完再开始写代码。在实现阶段，每完成一个页面截图给用户确认，确认后再写下一个。

> **关键洞察**：设计问题必须在编码前解决。一次性确认 30 分钟，远好过编码中迭代 2 小时。

# 6 第四次转变：工程化收尾

Day 6 之前，做了两轮工程化改造，解决的是"如何让这套流程可复制"的问题。

## 6.1 真实命令绑定

之前验证项目质量，每次都要手动敲一串命令。现在统一写进 Makefile：

```makefile
make verify           # 完整验证（后端 + 前端）
make backend-verify   # 后端：编译 + 测试 + 检查
make frontend-verify  # 前端：lint + 测试 + 构建
```

好处：
- AI 知道该执行什么命令
- 任何人都能重现相同结果
- CI/CD 可以直接调用

## 6.2 证据链管理

之前测试报告长这样：
```
✅ 测试通过
```

现在长这样：
```
执行命令：mvn clean test
返回码：0
执行时长：45 秒
测试通过：8/8
```

区别在于：后者是**可审计的**。出了问题能追溯，团队成员能验证。

## 6.3 经验独立化

踩过的坑怎么记录？

一开始写在一个线性的 `lessons-learned.md` 里，越写越长，找东西很困难。

后来改成独立文件，每个经验一个文件，带日期和主题：

```
docs/experiences/
├── 2026-03-25-frontend-visual-design-confirmation.md
├── 2026-03-26-quality-assurance-four-layers.md
└── 2026-03-26-documentation-reorganization.md
```

想找"前端设计"的经验？搜文件名就行。

## 6.4 文档目录演进

文档结构也经历了三次重组：

```
Day 1：  所有文档平铺在 docs/ 下
    ↓
Day 2：  按类型分目录（requirements/ architecture/ conventions/）
    ↓
Day 3+： 加 workflows/ 和 decisions/ 目录
    ↓
Harness整合：加 experiences/ 经验库，完成 7 层分类
```

最终的 7 层分类：

| 分类 | 放什么 | 多久更新 | 保留多久 |
| --- | --- | --- | --- |
| requirements/ | 需求和规划 | 频繁 | 长期 |
| architecture/ | 架构设计 | 中等 | 长期 |
| conventions/ | 编码规范 | 很少 | 长期 |
| workflows/ | 工作流程 | 很少 | 长期 |
| decisions/ | 架构决策记录（ADR） | 只写不改 | 永久 |
| experiences/ | 经验库 | 中等 | 长期 |
| reports/ | 执行报告 | 频繁 | 中期 |

> **关键洞察**：文档不是给人看的，是给 AI 看的。AI 的"记忆"全靠文件持久化——分类越清晰，AI 找到正确信息的概率越高。

# 7 最终的工作流

经过 4 次转变，最终形成了一个 7 阶段的自动化工作流：

```
用户说"实现 Day N"
       ↓
  ┌── 阶段 0: 准备
  │   检查入口、目录、前置状态
  │
  ├── 阶段 1: Plan Mode
  │   验证技术栈 → 设计数据库 → 定义 API → 前端设计确认
  │
  ├── 阶段 2: Implementation
  │   后端：实体 → DTO → Mapper → Service → Controller
  │   前端：读设计规范 → 实现 → 用户确认
  │
  ├── 阶段 3: Code Review
  │   前置检查 → git diff → Agent 审查 → 修复 Critical 问题
  │
  ├── 阶段 4: Testing
  │   前置检查 → 查文档 → 单元测试 → API 测试 → 证据链记录
  │
  ├── 阶段 5: Commit
  │   Conventional Commits 格式提交
  │
  └── 阶段 6: Reporting
      实施报告 → 经验更新 → 项目地图更新
```

每个阶段有明确的**进入条件、退出条件、失败处理**。整个流程用一个 Skill 文件驱动，AI 按文件执行，不需要人工盯。

# 8 踩坑总结

把 7 天踩过的坑分成四类：

## 技术选型的坑

| 坑 | 损失 | 教训 |
| --- | --- | --- |
| Spring Boot 3.2 + MyBatis-Plus 不兼容 | 30+ 分钟 | Plan Mode 必须验证版本兼容性 |
| JWT secret 长度不足 | 启动失败 | 配置项必须有检查清单 |
| Java 版本问题反复出现 | 每次手动设置 | 不要临时绕过，从根源解决 |

## 流程执行的坑

| 坑 | 损失 | 教训 |
| --- | --- | --- |
| AI 跳过 Plan Mode 直接写代码 | 返工 | 加入口检查：是否通过 Skill 进入 |
| AI 跳过 Code Review 直接测试 | 遗漏问题 | 加前置检查（3 秒） |
| 凭想象写测试脚本 | 反复调试 | 强制先获取真实 API 响应 |

## 设计迭代的坑

| 坑 | 损失 | 教训 |
| --- | --- | --- |
| 前端设计边写边改 | 2+ 小时、10+ 轮迭代 | 编码前一次性确认设计方案 |

## 工程化的坑

| 坑 | 损失 | 教训 |
| --- | --- | --- |
| 验证命令分散 | 新人不知道怎么验证 | Makefile 统一绑定 |
| 测试报告只写"通过" | 无法追溯 | 记录命令、返回码、时长 |
| 经验写在流水账里 | 找不到、用不上 | 独立文件 + 时间戳 + 主题 |

# 9 借鉴了哪些思想

| 思想 | 来源 | 怎么用的 |
| --- | --- | --- |
| Harness Engineering | OpenAI | 真实命令绑定、证据链管理、经验独立化 |
| ADR（架构决策记录） | Michael Nygard | 记录关键决策的背景、选项和理由，写了 3 个 ADR |
| 四层质量保障 | TDD + Code Review + CI/CD | 将测试和审查分离，定义触发条件 |
| Conventional Commits | 社区规范 | 统一 commit message 格式 |

**一个创新点**：把这些工程实践适配到了 AI 编程场景——核心区别是**所有流程必须文档化**，因为 AI 没有跨会话记忆，唯一的"记忆"就是文件系统。

# 11 八条核心原则

做完这个项目，如果只能记住八句话：

> **1. 流程优于天赋** — 好流程让普通 AI 产出稳定，没流程让强 AI 反复返工
>
> **2. 文档优于记忆** — AI 的记忆只能靠文件持久化，口头规则下个会话就忘
>
> **3. 前置优于事后** — Plan 阶段花 20 分钟验证，好过实施阶段花 2 小时返工
>
> **4. 自动优于手动** — 能被 Skill 执行的流程，绝不靠口头传达
>
> **5. 证据优于宣称** — 不是"测试通过了"，而是"返回码 0，8/8 通过，耗时 45s"
>
> **6. 强制优于自觉** — 靠 AI 自觉遵守流程永远不可靠，必须有检查点
>
> **7. 独立优于线性** — 经验要独立成文件，不要写在流水账里
>
> **8. 设计优于迭代** — 视觉方案先确认再编码，别边写边改

---

# 12 结论

AI 编程的真正挑战，不在于"能不能写出代码"——2026 年的大模型写代码已经够快了。

**真正的挑战在于：如何让 AI 在一个有质量要求的环境里，稳定地、可重复地完成工作。**

答案不是更强的模型，而是更好的工程：

- 文档化的流程（让 AI "记住"怎么做）
- 强制的检查点（让 AI "不得不"按流程走）
- 证据链的记录（让结果可追溯、可审计）
- 经验的沉淀（让同样的坑不踩第二次）

从 Day 1 的"直接写代码"到 Day 7 的"7 阶段自动化工作流"，这个项目走过的路，本质上就是软件工程几十年来走过的路——只不过这次，执行者换成了 AI。

> **核心不在于 AI 的能力有多强，而在于你给它搭的脚手架有多稳。**
