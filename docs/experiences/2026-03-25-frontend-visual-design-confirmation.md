# 前端视觉设计语言确认流程优化（2026-03-25）

## 问题场景

Day 5 前端页面（登录/注册）实现时，出现了大量的 UI 风格调整：
- 暗色调 → 亮色调 → 对比度调整 → 输入框样式 → 按钮交互 → 字体颜色
- 总共迭代 **10+ 轮**，每轮都需要用户在浏览器中查看效果后才能确认

**表现**：tech-design 阶段生成了技术方案，但没有确定设计语言，导致 implement-day 阶段反复决策。

## 根本原因

**缺少「视觉确认」环节**

前端开发与后端开发不同：
- 后端：遵循规范写代码 → 编译成功 → 测试通过 = 完成
- 前端：视觉设计需要在**设计阶段**一次确定，而不是在实现阶段重复决策

原有的 tech-design skill 没有强制前端设计语言确认，导致 implement-day 在每个页面实现时都要重新决策。

## 解决方案

### 1. 在 tech-design skill 中新增「前端视觉设计语言」步骤

**位置**：`skills/tech-design/SKILL.md` 第 163 行

**实现**：
- 在出方案前向用户确认 **4 个关键决策**：
  1. 色调风格（暗色系/亮色系/指定颜色）
  2. 品牌色（指定或 AI 决定）
  3. 视觉风格关键词（如「极简」「工业感」「编辑风」等）
  4. 参考设计（喜欢的网站/截图）

- 确认后，在方案文档中输出「前端设计规范」章节，包括：
  ```markdown
  ## 前端设计规范

  ### 设计语言
  - 风格方向：[如：工业极简 / 编辑风 / 清爽亮色]
  - 色调：[暗色系 / 亮色系]
  - 主色：[如：深墨棕 #1C1410]
  - 高亮色：[如：琥珀金 #D4870A]
  - 背景色：[如：暖奶油白 #FAF7F2]
  - 字体：标题 [字体名] / 正文 [字体名]

  ### 交互规范
  - 按钮默认态、hover 态、active 态
  - 输入框样式、focus 态、error 态
  - 错误提示、成功提示

  ### 约束
  - 禁止直接使用 UI 框架默认样式
  - 所有页面遵循以上设计语言，保持视觉一致性
  ```

- **关键约束**：这个章节一旦确定，后续所有前端页面的实现都必须遵守，不得重新决策。

### 2. 在 implement-day skill 中强制读取设计规范

**位置**：`skills/implement-day/SKILL.md` 第 215 行（1.5 前端页面强制规则）

**实现**：

**步骤 A：读取已定的设计规范**（不得跳过）
- 从技术方案文档（`.claude/plans/day-N-design.md`）中找「前端设计规范」章节
- 如果找不到 → **停止实现**，提示用户先用 tech-design skill 补充设计规范

**步骤 B：调用 frontend-design skill**
- 传递已确定的设计规范作为约束，**不允许自行重新决定风格**

**步骤 C：实现页面代码**
- 严格按设计规范实现
- 禁止使用 UI 框架默认样式直出页面

**步骤 D：实现完成后向用户展示效果确认**（必须）
- 实现完每个页面后，向用户说明：
  - 当前页面遵循的设计规范要点
  - 需要用户打开浏览器查看效果
  - 明确询问：「视觉效果是否符合预期？有需要调整的地方吗？」
  - **等待用户确认后再继续下一个页面**

### 3. 触发条件

**必须执行**（满足任意一条）：
- 任务描述中包含：页面、视图、组件、UI、界面、列表页、详情页、表单
- 文件路径包含：`views/`、`components/`、`pages/`
- 技术栈为 Vue / React / 纯前端

**不执行**：
- 纯后端任务（Java/Go/Python Service、Controller、Mapper 等）
- 仅修改配置文件、路由、Store（无视觉组件）

## 核心原则

> **设计语言在 tech-design 阶段一次确定，implement-day 阶段只执行不再重新决策。**

这个原则解决的核心问题：
- ❌ 错误做法：每个页面都问用户"你想要什么风格"
- ✅ 正确做法：一开始问一次"整体风格是什么"，然后所有页面都遵循这个决策

## 成功案例

Day 5 应用此规则后的效果：
- 设计阶段：在 tech-design 中确认"编辑风工业极简 + DM Serif Display + 琥珀金"
- 实现阶段：implement-day 读取设计规范 → frontend-design 按规范实现 → 用户确认 2 轮完成
- 时间对比：原来 10+ 轮调整 → 现在 1-2 轮确认

## 后续改进

可考虑的未来优化：
1. 在 tech-design 阶段生成设计稿（Figma/设计工具）而不仅是文字规范
2. 创建可复用的"设计预设"（模板），新项目可快速选择而不需要重新确认
3. 建立"设计规范检查清单"，implement-day 在完成页面后自动校验
