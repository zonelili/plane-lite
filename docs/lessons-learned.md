## 💡 Harness 实践经验

### 1. 项目初始化阶段

#### ✅ 做对了什么

**经验 1：先搭建完整的文档结构**
```
反模式：
❌ 直接开始写代码 → 需求不清晰 → 返工

最佳实践：
✅ 先完成 6 大核心文档 → 需求明确 → 减少返工

实例：
- 用 2 小时完成文档结构搭建
- 包含 86 页、27,000 字的详细文档
- 为后续开发提供清晰指引
```

**经验 2：CLAUDE.md 作为项目地图，不堆砌细节**
```
反模式：
❌ 把所有信息都塞进 CLAUDE.md → 超过 200 行 → AI 难以把握重点

最佳实践：
✅ CLAUDE.md 只作为"地图"，指向详细文档
✅ 采用渐进式披露原则

实例：
CLAUDE.md 结构：
- 项目概述（3 句话）
- 架构地图（指向 docs/）
- 快速开始（核心命令）
- 关键约束（5 条非协商规则）
```

**经验 3：定义"非协商规则"，约束 AI 行为**
```
反模式：
❌ 没有明确约束 → AI 随意发挥 → 代码风格不一致

最佳实践：
✅ 明确定义 5 条"非协商规则"
✅ 使用 MUST/NEVER 等强制性语言

实例：
1. 分层架构（Controller → Service → Mapper）
2. 数据库规范（表名小写、必须有时间戳）
3. API 设计（RESTful + 统一响应格式）
4. 前端规范（组件命名 PascalCase）
5. 代码质量（必须有事务、校验、测试）
```

#### ❌ 可以改进的地方

**教训 1：GitHub 认证方式选择**
```
遇到的问题：
- 尝试用 HTTPS 推送 → 需要输入用户名密码 → 失败
- 用户直接给了 Personal Access Token

更好的方式：
✅ 一开始就指导用户配置 SSH Key（一劳永逸）
✅ 或者先检查用户是否已有 SSH Key

改进措施：
- 下次项目初始化时，优先推荐 SSH 方式
- 提供 SSH Key 配置的详细步骤
```

## 🤝 人机协作经验

### 1. 需求澄清阶段

#### 经验 1：用户需求可能不明确，要主动提问

**场景**：
```
用户："我想根据 PQCP 的 PRD 重新写一个系统"
问题：没有初始 PRD，只有 2 期 PRD
```

**处理方式**：
```
✅ 主动提出问题：
  - "没有初始 PRD 会导致业务理解不完整"
  - 提供 4 个替代方案让用户选择

结果：
  - 用户选择了更合适的方案（模仿 Plane）
  - 避免了基于不完整需求开发
```

**可复用模式**：
```
当遇到需求不明确时：
1. 明确指出问题所在
2. 提供 2-4 个替代方案
3. 每个方案说明优劣势
4. 让用户做选择题（而非开放题）
```

#### 经验 2：用户可能不了解原型项目，需要先建立认知

**场景**：
```
用户："我其实还不了解这个开源项目"
```

**处理方式**：
```
✅ 立即暂停开发，先建立认知：
  1. 深入解读原项目（是什么、解决什么问题）
  2. 核心概念解析（Workspace、Project、Issue、Cycle）
  3. 功能演示（看板、列表、甘特图）
  4. 竞品对比（vs Jira、Linear）
  5. 建议先体验后开发

结果：
  - 用户决定先去 plane.so 体验
  - 有利于后续开发时做更好的简化决策
```

**可复用模式**：
```
当模仿开源项目时：
1. 先确认用户是否了解原项目
2. 如果不了解，提供全面解读
3. 建议用户先体验原项目（15 分钟）
4. 体验后再决定哪些功能要做、哪些要简化
```

### 2. 工具使用阶段

#### 经验 3：GitHub 仓库创建要细致指导

**场景**：
```
用户："你可以我给你账号密码你去创建吗"
```

**处理方式**：
```
❌ 不能接收用户密码（安全问题）
✅ 提供详细的操作指导：
  - 3 个创建入口的位置
  - 每个步骤的截图说明
  - 需要填写的具体信息
  - 不需要勾选的选项（避免冲突）

结果：
  - 用户成功创建仓库
  - 顺利配置 Personal Access Token
```

**可复用模式**：
```
当指导用户操作时：
1. 明确拒绝不安全的操作（接收密码）
2. 提供多个操作入口（适应不同界面版本）
3. 详细说明每个步骤
4. 特别强调"不要做什么"（避免错误）
```

## 🛠️ 技术决策经验

### 1. 技术栈选择

#### 决策 1：为什么不用原项目的技术栈？

**原项目技术栈**：Django + Next.js + PostgreSQL

**我们的选择**：Spring Boot + Vue 3 + MySQL

**理由**：
```
✅ 团队技术栈：Java 是主技术栈
✅ 学习曲线：Vue 3 比 Next.js 简单
✅ 运维成本：MySQL 比 PostgreSQL 更熟悉
✅ AI 训练数据：Spring Boot 的训练数据更丰富
```

**经验总结**：
```
技术选型原则：
1. 优先考虑团队技术栈（减少学习成本）
2. 选择 AI 更熟悉的技术（提高代码质量）
3. 选择运维成本低的技术（便于维护）
4. 功能实现优先，不追求新技术
```

### 2. 架构设计

#### 决策 2：MVP 功能范围如何确定？

**初始想法**：实现 Plane 的所有核心功能

**最终决策**：只做 5 个核心模块

**取舍逻辑**：
```
✅ 必须做（MVP）：
  - 用户认证（注册、登录）
  - 项目管理（CRUD）
  - 问题管理（CRUD + 状态流转）
  - 看板视图（基础展示）
  - 评论系统（基础功能）

❌ 暂不做（V2.0+）：
  - 拖拽操作（V2.0）
  - 标签系统（V2.0）
  - 冲刺管理（V3.0）
  - 富文本编辑器（V3.0）
  - 文件上传（V3.0）
  - 统计分析（V4.0）
```

**经验总结**：
```
MVP 范围确定原则：
1. 核心流程可用（注册 → 创建项目 → 管理问题 → 看板查看）
2. 功能简化不简陋（基础功能完整）
3. 技术债务可控（不做过度设计）
4. 1 周能完成（时间盒约束）
```

## 📝 文档编写经验

### 1. 文档结构设计

#### 经验 1：分层文档 vs 单一文档

**反模式**：
```
❌ 所有内容写在一个大文档里
  - 文档超过 100 页
  - 难以维护
  - AI 难以定位信息
```

**最佳实践**：
```
✅ 分层文档结构：
  requirements/ - 需求文档
  architecture/ - 架构设计
  conventions/ - 编码规范
  workflows/ - 工作流程
  decisions/ - 架构决策记录

优势：
  - 职责清晰
  - 易于更新
  - AI 可以精准读取需要的部分
```

### 2. 文档内容质量

#### 经验 2：提供具体示例而非空洞理论

**反模式**：
```
❌ "API 接口应该遵循 RESTful 规范"
  - 太抽象
  - 开发者不知道具体怎么做
```

**最佳实践**：
```
✅ 提供完整的请求/响应示例：

POST /api/v1/issues

Request:
{
  "project_id": 1,
  "title": "实现用户登录功能",
  "priority": "high"
}

Response:
{
  "code": 200,
  "message": "创建成功",
  "data": {
    "id": 1,
    "issue_number": "PL-1",
    "title": "实现用户登录功能"
  }
}

优势：
  - 开发者一看就懂
  - AI 可以直接复制模式
  - 减少理解偏差
```

## 🔄 迭代开发经验

### 1. 开发节奏

#### 经验 1：先体验再开发 vs 直接开发

**场景**：
```
用户："我其实还不了解这个开源项目"
```

**决策**：
```
✅ 建议先体验 15 分钟，再开始开发

理由：
1. 避免盲目开发（不知道产品长什么样）
2. 更好的简化决策（知道哪些功能可以砍）
3. 更好的用户体验（知道好的交互是什么样的）
```

**经验总结**：
```
模仿开源项目的开发流程：
1. 深入了解原项目（15 分钟体验）
2. 分析核心功能（哪些必做、哪些可选）
3. 制定简化方案（MVP 范围）
4. 开始开发
```
