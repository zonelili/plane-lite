## ⚠️ 质量保障机制的教训（Day 2 + 更新）

### 暴露的问题

在 Day 2 开发过程中，我们遇到了三类质量问题。后续分析发现，当前的三层机制**缺少独立的测试环节**。

#### 问题 1：版本兼容性问题

**现象**：
```
Spring Boot 3.2 + MyBatis-Plus 3.5.5 不兼容
反复犯同样的错误，浪费了大量时间
```

**根本原因**：
- 技术选型时没有查阅官方兼容性文档
- 没有查看 GitHub issues 中的已知问题
- 依赖直觉判断"新版本应该兼容"

**损失**：
- 浪费 30+ 分钟排查问题
- 需要回滚版本并重新测试
- 影响开发进度

#### 问题 2：代码与方案不一致

**现象**：
```
Project 实体类有 leadId 字段
但数据库脚本 V1__init.sql 中缺少 lead_id 列
运行时报错：Unknown column 'lead_id' in 'field list'
```

**根本原因**：
- 编写代码时没有同步更新数据库脚本
- 没有验证实体类与数据库表的一致性
- 缺少代码审查机制

**损失**：
- 启动失败，需要人工干预
- 需要修改数据库脚本并重新启动
- 如果在生产环境会导致严重问题

#### 问题 3：环境配置问题

**现象**：
```
MySQL 密码使用了默认值 'password'，未更新为实际密码
JWT secret 长度为 55 字符（440 bits），不满足 HS512 要求（>= 512 bits）
启动失败，需要人工修改配置文件
```

**根本原因**：
- 配置文件使用默认值，没有验证是否符合要求
- 没有环境配置预检查机制
- 启动失败才被动修复，而不是主动验证

**损失**：
- 每次启动都需要检查配置文件
- 增加了开发环境搭建的复杂度
- 容易遗漏配置项

### 根本原因分析

**核心问题**：缺乏系统化的质量保障机制

```
反模式：
❌ 方案设计时：没有验证技术栈兼容性和环境配置
❌ 编码过程中：写完代码直接提交，没有自检
❌ 提交代码前：没有审查代码与 Plan 的一致性
❌ 流程记忆：没有文档化，AI 每次都需要重新思考
```

**为什么会这样？**
1. **缺乏方案设计阶段的验证**：技术选型靠直觉，不查文档
2. **缺乏代码审查机制**：编码完成后直接提交，不对比 Plan
3. **缺乏环境预检查**：启动失败才修复，不主动验证
4. **流程不持久**：工作流程没有文档化，AI 无法记忆

### 解决方案：四层质量保障机制

建立**自动化的四层质量保障机制**，将质量检查**持久化到文档**中：

#### Layer 1: Plan Mode（方案设计）

**触发条件**：新功能开发、技术选型变更、数据库设计

**必须包含**：
- ✅ 技术栈兼容性验证（查阅官方文档，避免版本冲突）
- ✅ 环境依赖清单（MySQL、Redis、JWT 配置要求）
- ✅ 完整的 CREATE TABLE DDL（实体类字段与数据库列一一对应）
- ✅ 验收标准（编译、启动、API 测试）

**详见**：`/docs/workflows/quality-assurance.md#layer-1`

#### Layer 2: Implementation（编码实现）

**编码中自检**：
- ✅ 实体类 ↔ 数据库表一致性（字段名、类型、注解）
- ✅ DTO 校验注解（`@NotNull`, `@NotBlank`, `@Size`）
- ✅ Service 事务注解（写操作 `@Transactional`）
- ✅ Controller Swagger 文档（`@Tag`, `@Operation`）

**配置检查**：
- ✅ 密码长度、连接信息、端口冲突

**详见**：`/docs/workflows/quality-assurance.md#layer-2`

#### Layer 3: Testing（测试验证）

**触发条件**：完成一个 Day 的任务、Bug 修复后、准备 Code Review 前

**测试步骤**：
1. **单元测试验证**：mvn test（后端）/ vue skill（前端）
2. **API 功能测试**：手动测试（Postman/curl）
3. **部署验证**：启动服务，健康检查

**验收标准**：
- ✅ 所有单元测试通过
- ⚠️ 测试覆盖率建议 >= 80%（不强制）
- ✅ 所有 API 端点手动测试通过
- ✅ 服务启动成功，健康检查通过

**详见**：`/docs/workflows/quality-assurance.md#layer-3`

#### Layer 4: Code Review（代码审查）

**触发条件**：Testing 通过后、准备 commit

**使用工具**：`superpowers:requesting-code-review` skill

**检查内容**：
- ✅ 代码 vs Plan 一致性（API、字段、技术栈版本）
- ✅ 实体类 vs 数据库表一致性（字段名转换、类型匹配）
- ✅ 配置文件完整性（MySQL、JWT、Redis 配置验证）
- ✅ 代码规范和最佳实践

**详见**：`/docs/workflows/quality-assurance.md#layer-4`

### 为什么要增加测试层？

**问题背景**：
用户指出当前的三层机制缺少独立的测试环节。在传统开发团队中，流程应该是：
```
开发完成 → 测试团队测试 → 代码审查 → 提交
```

但我们的三层机制将测试和审查混在一起（Layer 3: Code Review 中包含编译、启动、API 测试），导致：
- **职责不清**：测试验证（功能是否正常）和代码审查（代码是否符合设计）混在一起
- **错误发现延迟**：功能 Bug 应该在测试阶段发现，而不是在审查阶段
- **流程不符合直觉**：违反了"测试团队独立验证"的传统流程

**改进方案**：
- 将测试验证独立为 Layer 3: Testing
- 将代码审查改为 Layer 4: Code Review
- Testing 重点验证功能是否正常（单元测试、API 测试、部署验证）
- Code Review 重点验证代码是否符合设计（代码 vs Plan、实体 vs 数据库、代码规范）

### 关键原则

#### 1. 记忆持久化

**原则**：工作流程必须写入文档，AI 才能记住和执行

**反例**：❌ 口头告知 AI "记住要检查版本兼容性"（下次就忘了）

**正例**：✅ 在文档中写明检查清单和验证方式

**实施**：
- 创建 `/docs/workflows/quality-assurance.md` 详细文档
- 在 `CLAUDE.md` 中引用质量保障流程
- 创建 ADR-001 记录设计决策

#### 2. 自动化触发

**原则**：定义明确的触发条件，避免遗漏

**触发规则**：
- 用户说"开始 Day N"时 → 自动进入 Plan Mode
- 完成模块开发后 → 自动触发 Code Review
- 修复 Bug 后 → 自动验证测试通过

#### 3. 检查清单化

**原则**：可验证、可追溯的检查清单

**好的检查清单特征**：
- ✅ 明确的检查项（不是模糊的"检查代码质量"）
- ✅ 可量化的标准（JWT secret >= 64 chars）
- ✅ 具体的验证方式（读取文件、执行命令）
- ✅ 清晰的输出格式（✅/❌/⚠️ 状态）

### 实施成果

**文档化**：
- ✅ 创建 `/docs/workflows/quality-assurance.md` - 质量保障流程详细文档
- ✅ 更新 `CLAUDE.md` - 添加质量保障流程说明
- ✅ 创建 `ADR-001` - 架构决策记录
- ✅ 修复 `V1__init.sql` - 添加缺失的 `lead_id` 列

**预期效果**：
- ✅ Day 3 开发时自动触发 Plan Mode
- ✅ 代码完成后自动触发 Code Review
- ✅ 不再出现"实体类与数据库不一致"的问题
- ✅ 不再出现"版本不兼容"的问题

### 可复用模式

**遇到质量问题时的处理流程**：
```
1. 记录问题现象（具体的错误信息）
2. 分析根本原因（为什么会发生）
3. 评估损失（浪费的时间和影响范围）
4. 设计预防机制（如何避免再次发生）
5. 文档化流程（写入文档，持久化记忆）
6. 验证有效性（下次开发时检验）
```

**质量保障机制的设计原则**：
```
1. 前置验证（Plan Mode 阶段检查技术栈兼容性）
2. 过程自检（Implementation 阶段实时检查）
3. 独立测试（Testing 阶段验证功能正常）
4. 提交审查（Code Review 阶段验证代码符合设计）
5. 文档持久（写入文档，AI 可以记忆）
6. 自动触发（明确触发条件，避免遗漏）
```

## 💡 Harness 实践经验

### 1. 项目初始化阶段

#### ✅ 做对了什么

**经验 1：先搭建完整的文档结构**
```
反模式：
❌ 直接开始写代码 → 需求不清晰 → 返工

最佳实践：
✅ 先完成 6 大核心文档 → 需求明确 → 减少返工

实例：
- 用 2 小时完成文档结构搭建
- 包含 86 页、27,000 字的详细文档
- 为后续开发提供清晰指引
```

**经验 2：CLAUDE.md 作为项目地图，不堆砌细节**
```
反模式：
❌ 把所有信息都塞进 CLAUDE.md → 超过 200 行 → AI 难以把握重点

最佳实践：
✅ CLAUDE.md 只作为"地图"，指向详细文档
✅ 采用渐进式披露原则

实例：
CLAUDE.md 结构：
- 项目概述（3 句话）
- 架构地图（指向 docs/）
- 快速开始（核心命令）
- 关键约束（5 条非协商规则）
```

**经验 3：CLAUDE.md 必须是完整的项目地图,不能省略任何目录**

反模式：
❌ CLAUDE.md 只列出主要目录 → AI 找不到具体文件
❌ 把 CLAUDE.md 当成"开发指南" → 与 README.md 职责混淆
❌ 担心"堆砌细节"而省略索引 → 失去了地图的作用
❌ README.md 和 CLAUDE.md 内容重复 → 维护成本高

最佳实践：
✅ CLAUDE.md = 项目地图 = 所有目录和文件的索引
✅ 每个目录都有详细说明,每个文件都有一行用途
✅ 使用代码块展示目录树(清晰易读)
✅ 标注空目录状态(待实现)
✅ 删除 README.md,避免职责混淆

正确理解：
- 地图 ≠ 简略,地图 = 全面 + 简洁
- CLAUDE.md(给 AI): 项目地图,快速定位任何文件
- README.md(给人): 项目介绍,吸引用户和开发者 → 在本项目中已删除,避免重复
- docs/README.md(给深入者): 文档索引和分类规范

实例：
正确的 CLAUDE.md 结构：
1. 列出根目录的所有重要文件
2. 列出所有一级目录
3. /backend/ 详细到包级别(common/, config/, module/)
4. 每个业务模块详细到子目录(controller/, dto/, entity/, mapper/, service/)
5. 配置文件和数据库脚本列出
6. 空目录标注"⚠️ 待实现"并显示计划结构
7. /docs/ 显示完整的子目录树

教训：
给自己加的限制太多,过度害怕"堆砌细节",导致矫枉过正。
应该是"全面索引 + 一行说明",而不是"省略索引"。
**文档职责必须清晰,不能重复写或混淆内容**。

**经验 4：文档分类必须清晰,CLAUDE.md 索引必须及时更新**
```
反模式：
❌ 文档随意放在根目录 → 结构混乱 → 难以查找
❌ 添加新文档后不更新 CLAUDE.md → 索引失效 → AI 找不到
❌ 不区分持久文档和临时文档 → 文档堆积 → 难以维护

最佳实践：
✅ 按功能分类文档目录（requirements, architecture, conventions）
✅ 临时文档独立目录（reports/）
✅ 每次添加新文档立即更新 CLAUDE.md 索引
✅ 统一命名规范（小写字母 + 连字符）

实例：
正确的文档结构：
docs/
├── requirements/       # 需求文档（持久）
├── architecture/       # 架构设计（持久）
├── conventions/        # 编码规范（持久）
├── workflows/          # 工作流程（持久）
├── decisions/          # ADR（持久）
├── reports/            # 审查报告（临时）✨
└── README.md           # 详细索引

文档创建前必须问：
1. 这是持久文档还是临时文档？
2. 应该放在哪个目录？
3. 文档命名是否符合规范？
4. CLAUDE.md 是否需要更新索引？

详见：/docs/reports/documentation-reorganization.md
```

**经验 5：定义"非协商规则"，约束 AI 行为**
```
反模式：
❌ 没有明确约束 → AI 随意发挥 → 代码风格不一致

最佳实践：
✅ 明确定义 5 条"非协商规则"
✅ 使用 MUST/NEVER 等强制性语言

实例：
1. 分层架构（Controller → Service → Mapper）
2. 数据库规范（表名小写、必须有时间戳）
3. API 设计（RESTful + 统一响应格式）
4. 前端规范（组件命名 PascalCase）
5. 代码质量（必须有事务、校验、测试）
```

#### ❌ 可以改进的地方

## 🤝 人机协作经验

### 1. 需求澄清阶段

#### 经验 1：用户需求可能不明确，要主动提问

**场景**：
```
用户："我想根据 PQCP 的 PRD 重新写一个系统"
问题：没有初始 PRD，只有 2 期 PRD
```

**处理方式**：
```
✅ 主动提出问题：
  - "没有初始 PRD 会导致业务理解不完整"
  - 提供 4 个替代方案让用户选择

结果：
  - 用户选择了更合适的方案（模仿 Plane）
  - 避免了基于不完整需求开发
```

**可复用模式**：
```
当遇到需求不明确时：
1. 明确指出问题所在
2. 提供 2-4 个替代方案
3. 每个方案说明优劣势
4. 让用户做选择题（而非开放题）
```

#### 经验 2：用户可能不了解原型项目，需要先建立认知

**场景**：
```
用户："我其实还不了解这个开源项目"
```

**处理方式**：
```
✅ 立即暂停开发，先建立认知：
  1. 深入解读原项目（是什么、解决什么问题）
  2. 核心概念解析（Workspace、Project、Issue、Cycle）
  3. 功能演示（看板、列表、甘特图）
  4. 竞品对比（vs Jira、Linear）
  5. 建议先体验后开发

结果：
  - 用户决定先去 plane.so 体验
  - 有利于后续开发时做更好的简化决策
```

**可复用模式**：
```
当模仿开源项目时：
1. 先确认用户是否了解原项目
2. 如果不了解，提供全面解读
3. 建议用户先体验原项目（15 分钟）
4. 体验后再决定哪些功能要做、哪些要简化
```

### 2. 工具使用阶段

#### 经验 3：GitHub 仓库创建要细致指导

**场景**：
```
用户："你可以我给你账号密码你去创建吗"
```

**处理方式**：
```
❌ 不能接收用户密码（安全问题）
✅ 提供详细的操作指导：
  - 3 个创建入口的位置
  - 每个步骤的截图说明
  - 需要填写的具体信息
  - 不需要勾选的选项（避免冲突）

结果：
  - 用户成功创建仓库
  - 顺利配置 Personal Access Token
```

**可复用模式**：
```
当指导用户操作时：
1. 明确拒绝不安全的操作（接收密码）
2. 提供多个操作入口（适应不同界面版本）
3. 详细说明每个步骤
4. 特别强调"不要做什么"（避免错误）
```

## 🛠️ 技术决策经验

### 1. 技术栈选择

#### 决策 1：为什么不用原项目的技术栈？

**原项目技术栈**：Django + Next.js + PostgreSQL

**我们的选择**：Spring Boot + Vue 3 + MySQL

**理由**：
```
✅ 团队技术栈：Java 是主技术栈
✅ 学习曲线：Vue 3 比 Next.js 简单

```

**经验总结**：
```
技术选型原则：
1. 优先考虑团队技术栈（减少学习成本）

4. 功能实现优先，不追求新技术
```

### 4. 核心技术实现亮点

#### 4.1 sequence_id 并发安全方案

**问题**：如何保证 `sequence_id` 在高并发下不重复？

**解决方案**：
```java
// 1. 数据库唯一索引
UNIQUE KEY `uk_project_sequence` (`project_id`, `sequence_id`)

// 2. Service 层重试机制
try {
    Integer maxSequenceId = issueMapper.selectMaxSequenceIdByProject(projectId);
    Integer nextSequenceId = (maxSequenceId == null) ? 1 : maxSequenceId + 1;
    issueMapper.insert(issue);
} catch (DuplicateKeyException e) {
    retryCount++;
    // 最多重试 3 次
}
```

**测试结果**：
- 连续快速创建 3 个问题，无冲突
- 问题编号：TEST-1, TEST-2, TEST-3（完美递增）

**经验教训**：
> "并发安全 = 数据库约束（最终一致性）+ 应用层重试（处理冲突）"

1. **不要临时绕过，要从根源解决**

   - 反例：要求用户手动在 Swagger UI 中测试

4. **Code Review 不是可选的，是必须的**
   - 价值：发现了 Project 实体缺失字段的问题
   - 结论：避免了更大的质量问题

5. **过早优化是万恶之源**
   - 问题编号动态生成：性能损失可接受
   - 结论：先实现功能，瓶颈出现时再优化

6. **文档不是负担，是资产**
   - 完整的文档让项目可维护、可传承
   - 测试报告可重复执行

7. **多项目环境下的目录管理（Day 4，2026-03-24）**
   - **现象**：当 `~/Documents/myproject/` 下有多个项目（如 `qpp-api` 和 `plane-lite`）时，如果不明确工作目录，AI 容易在错误的项目中操作（"目录踩踏"）
   - **原因**：相对路径假设 + 缺少工作目录验证
   - **解决方案**：
     - **用户侧**：提供任务计划时，明确说明项目路径或先用 `cd` 切换到项目根目录
     - **AI 侧**：收到任务后第一步用 `pwd` 确认当前目录，发现不对立即切换
   - **教训**：不要假设路径，先验证再操作

8. **测试前必须先查文档（Day 4，2026-03-24）**
   - **现象**：编写测试脚本时反复犯同样的错误
     - Day 3 测试：可能犯过登录接口错误
     - Day 4 测试：又用 `username` 而不是 `email` 登录，Token 提取逻辑也写错
     - 根本原因：凭记忆/猜测 API 格式，没有查文档；**凭想象写提取逻辑，没有基于真实响应验证**
   - **问题根源**：
     - ✅ 项目有 API 文档（`docs/conventions/api.md`），明确写了登录示例
     - ✅ 有历史测试报告（`day3-test-results.md`），可以复用逻辑
     - ❌ 但 AI 没有养成"先查文档"的习惯，直接开始写测试
     - ❌ **更严重：看了文档后，凭想象写 `grep` 表达式，不先获取真实响应验证**
   - **解决方案**：
     - **改进 implement-day skill（第 1 次）**：在测试阶段增加"步骤 0：查阅文档"
       1. API 文档（`docs/conventions/api.md`）- 接口格式、认证方式
       2. 历史测试报告（`docs/reports/day-{N-1}-test-results.md`）- 复用测试逻辑
       3. 数据库脚本（`db/migration/V1__init.sql`）- 测试用户凭据
     - **改进 implement-day skill（第 2 次）**：在测试阶段强制"先获取真实响应，再写提取逻辑"
       - 步骤 2.1：先手动调用 API，获取真实 JSON 响应
       - 步骤 2.2：基于真实响应编写提取逻辑（不凭想象）
       - 步骤 2.3：逐个 API 测试并验证（不写完整脚本再运行）
       - 步骤 2.4：保存测试脚本供后续复用
   - **教训**：
     - **不要凭记忆，先查文档**：尤其是登录接口、认证方式这种容易变化的部分
     - **不要凭想象，先看真实响应**：写任何提取逻辑前，先 `curl ... | jq .` 看真实数据
     - **边写边验证，不要写完再测**：一次写一个 API 调用，立即验证，再写下一个
     - **保存测试脚本**：Day N+1 必须能复用 Day N 的逻辑
     - **Skill 需要强制检查点**：加入"查文档"和"先获取真实响应"作为必须步骤

9. **缺少自动检查机制导致跳过流程（Day 4，2026-03-24）**
   - **现象**：Day 4 执行时完全违反了质量保障流程
     - ❌ 没有调用 `implement-day` skill（应该在开始时触发）
     - ❌ 编译后直接测试，跳过了 Code Review
     - ✅ 测试阶段遵守了新流程（基于真实响应）
     - 实际流程：`Implementation → 编译 → 测试 → 才想起来 Code Review`
     - 应该流程：`Plan → Implementation → Code Review → Testing → Commit`
   - **问题根源**：
     - 用户给了"实施计划"，AI 就直接执行，没有意识到应该通过 skill 来执行
     - 编译通过后的惯性思维："好！可以测试了" → 完全忘记中间还有 Code Review
     - **根本问题：缺少自动检查机制**，靠 AI "自觉"永远不可靠
   - **解决方案**：
     - **改进 implement-day skill（第 3 次）**：在每个阶段开头增加强制前置检查
       - 阶段 0：工作流自检（5秒）- 确认是通过 skill 进入、目录正确、无未完成阶段
       - 阶段 3：Code Review 前置检查（3秒）- 必须 Implementation 完成且编译通过
       - 阶段 4：Testing 前置检查（3秒）- **必须 Code Review 完成**（最容易跳过）
     - **总时间成本**：< 20 秒/每个 Day，远低于返工成本（30+ 分钟）
   - **教训**：
     - **不能依赖 AI 自觉**：即使 skill 写得再好，如果不强制检查，还是会忘记
     - **前置检查必须简单快速**：< 3 秒的检查不会浪费时间，但能避免大问题
     - **失败必须立即停止**：检查不通过时，不允许继续，强制回到正确流程
     - **Skill 是流程守护者**：通过强制检查点确保流程不被跳过

10. **Code Review 只审查变动部分，节省 token（Day 4，2026-03-24）**
   - **问题**：Code Review Agent 每次读取所有文件，消耗大量 token
   - **优化方案**：只审查变动部分（`git diff HEAD` 或 `git diff BASE_SHA HEAD_SHA`）
   - **效果**：
     - 新增模块：差别不大（所有文件都是新的）
     - 修改已有代码：token 消耗大幅降低（只读取修改的行）
   - **实施**：
     - 更新 `implement-day` skill 阶段3（Code Review）
     - 使用 `git diff` 命令获取变动，传递给 code-reviewer agent
   - **教训**：工作流程要持续优化，不是"记在心里"，而是**写入流程文档**

### 核心原则总结

#### 1. 永远使用 Skill 工具

**红线规则**：
- ✅ 看到"实现 Day X"、"开始 Day X" → **立即调用 implement-day skill**
- ✅ 看到"修复 bug"、"调试问题" → 考虑使用 `systematic-debugging` skill
- ✅ 看到"审查代码"、"review" → 使用 `code-review` skill

**不要这样想**：
- ❌ "这个任务很简单，不需要 skill"
- ❌ "我记得 skill 的内容，不需要调用"
- ❌ "先快速写代码，再补 skill"

**正确思维**：
- ✅ **Skill 是规范，不是建议**
- ✅ **即使 1% 可能适用，也要调用**
- ✅ **Skill 调用成本 < 5秒，返工成本 > 30分钟**

#### 2. 严格执行阶段顺序

**不可颠倒的顺序**：
```
Code Review（先）→ Testing（后）
```

**原因**：
1. Code Review 发现的是**代码质量问题**（逻辑、规范、一致性）
2. Testing 发现的是**功能正确性问题**
3. 先保证代码质量，再验证功能，效率最高

#### 3. 测试前必须查文档

**查文档的优先级**：
1. **API 文档**（必读）- 了解请求/响应格式
2. **历史测试报告**（优先复用）- 复用已验证的逻辑
3. **数据库初始化脚本**（如需测试用户）- 确认测试凭据

**查文档后的步骤**：
1. 手动调用 API 看真实响应
2. 基于真实响应写提取逻辑
3. 逐个 API 测试并验证
4. 保存测试脚本供 Day N+1 复用

#### 4. 前置检查只需 5 秒

**阶段 0 的 3 个问题**（必须回答）：
1. 我是通过 skill 进入的吗？
2. 当前工作目录正确吗？
3. 是否有未完成的前置阶段？

**时间成本**：< 5 秒
**避免的返工成本**：> 30 分钟

## 🎨 Day 5 前端视觉迭代教训（2026-03-25）

- **implement-day skill**：Step 1.5 改为先读取方案中的设计规范，找不到则停止；实现完每个页面后向用户展示效果确认，通过后再继续

### 核心原则

> 设计语言在 tech-design 阶段一次确定，implement-day 阶段只执行不再重新决策。
