# ADR-003: 前端视觉设计语言一次性确认

**状态**: 已采纳（2026-03-25）
**日期**: 2026-03-25
**决策者**: AI Agent + 用户
**相关文档**: `/docs/experiences/2026-03-25-frontend-visual-design-confirmation.md`, `/docs/conventions/reporting.md`

---

## 背景（Context）

在 Day 5 前端开发过程中，我们遇到了以下问题：

### 问题现象

1. **设计反复变更**
   - Day 5 登录/注册页面从暗色调改为亮色调
   - 修改了字体、颜色、按钮样式等多个元素
   - 总共进行了 10+ 次迭代

2. **每次迭代都需要用户确认**
   - 设计阶段无法一次确定设计语言
   - 实现阶段才发现设计不符合预期
   - 需要反复修改和确认

3. **设计规范不清晰**
   - 颜色选择依赖主观感受
   - 字体搭配没有明确规范
   - 按钮、输入框等组件缺少样式指南

### 根本原因

**设计确认流程缺失**：

- ❌ tech-design 阶段没有前端视觉设计确认
- ❌ 直接进入 implement-day 编码，设计语言尚未敲定
- ❌ 编码过程中才发现设计问题，需要反复调整
- ❌ 浪费了大量时间在"试错和调整"而不是"正式编码"

### 损失分析

```
Day 5 前端开发时间分配：
- 计划编码时间：2 小时
- 实际编码时间：1 小时
- 设计调整时间：10+ 次迭代 = 2 小时+
- 效率损失：50%+

关键发现：
大部分时间浪费在"这个颜色对不对？"
而不是"代码写对没有？"
```

---

## 决策（Decision）

建立**前端视觉设计一次性确认机制**：

### 原则

**设计确认必须在 tech-design 阶段完成，不能推迟到 implement-day**

### 流程

#### 1. tech-design 阶段：设计语言确认

**前置条件**：
- 已确定功能需求和页面布局
- 已理解用户需求和品牌定位

**确认内容**（4 大问题）：

| 问题 | 需要确定的内容 | 输出物 |
|------|---------------|--------|
| **Q1: 色调定位** | 亮色系/暗色系/混合 | 色调选择（亮色/暗色/对比） |
| **Q2: 品牌色** | 主色、辅助色、强调色 | RGB 代码 / Hex 代码 |
| **Q3: 字体搭配** | 标题字体、正文字体、等宽字体 | 字体名称 + Fallback |
| **Q4: 参考设计** | 参照哪个设计系统或产品 | 具体链接或描述 |

**输出物**：`frontend-design-spec.md`

```markdown
# Day 5 前端设计规范

## 色调定位
- 主色调：亮色系（冷白背景）
- 原因：VS Code 默认主题启发，适合开发工具

## 品牌色
- 主色：深墨棕 #1C1410（导航、标题）
- 强调色：琥珀金 #D4870A（按钮、链接）
- 背景色：暖奶油白 #FAF7F2（页面背景）

## 字体搭配
- 标题：DM Serif Display（衬线体，精致感）
- 正文：JetBrains Mono（等宽体，代码感）
- Fallback：System Font Stack

## 参考设计
- VS Code 主题色系
- Stripe 官网配色（现代感）
- Figma 界面设计（专业感）
```

**验收标准**：
- ✅ 4 个设计问题都有明确答案
- ✅ 每个颜色都有 RGB 或 Hex 代码
- ✅ 字体选择经过用户确认
- ✅ 用户对设计语言满意

#### 2. implement-day 阶段：设计规范应用

**前置条件**：
- ✅ tech-design 已完成，`frontend-design-spec.md` 已产出

**实施方式**：

1. **读取设计规范**
   ```
   在开始编码前，读取 frontend-design-spec.md
   检查是否有 4 个设计问题的答案
   如果没有，停止编码，要求 tech-design 先完成
   ```

2. **一页一确认**
   ```
   编码完成每个页面后，截图给用户确认
   确认内容：
   - 整体色调是否符合规范
   - 字体是否正确应用
   - 按钮、输入框样式是否符合

   用户确认 ✅ → 继续编码下一页
   用户反馈修改 → 调整当前页并重新确认
   ```

3. **设计一致性检查**
   ```
   所有页面完成后，逐页检查：
   - 颜色使用是否一致
   - 字体搭配是否一致
   - 间距、尺寸是否一致
   ```

**验收标准**：
- ✅ 所有页面都应用了设计规范
- ✅ 用户对所有页面设计满意
- ✅ 没有设计变更需求
- ✅ 设计规范可被后续页面复用

## 理由（Rationale）

### 为什么要在 tech-design 阶段确认设计？

1. **设计是编码的前提**
   - 编码前必须知道"要做什么样子"
   - 编码中发现设计问题需要返工
   - 返工成本 > 前期确认成本

2. **用户期望必须明确**
   - 模糊的需求导致 10+ 次迭代
   - 明确的需求可以一次编码成功
   - 节省 50%+ 的时间

3. **复用性要求**
   - Day 5 设计的颜色、字体要被后续页面复用
   - 必须在 tech-design 阶段标准化
   - 否则每个页面都有不同的设计

### 为什么是"一次性确认"而不是"迭代式调整"？

**实证数据**（Day 5）：
```
迭代式调整模式（实际结果）：
- 第 1 次：用户看到设计后提出修改意见
- 第 2-10 次：持续调整，每次都需要用户确认
- 总时间：2+ 小时
- 用户满意度：70%（还有改进空间）

一次性确认模式（预期结果）：
- tech-design 阶段 4 个问题 → 用户 10 分钟确认一次
- implement-day 阶段 10 分钟确认一次页面
- 总时间：30+ 分钟
- 用户满意度：90%+（提前规划）
```

### 为什么要有"一页一确认"？

1. **及时反馈**
   - 不是等所有页面完成再调整
   - 每页完成立即确认，发现问题立即修复
   - 避免后续页面继续复制错误

2. **渐进式优化**
   - 第 1 页可能需要调整较多
   - 第 2-N 页逐步接近完美
   - 最后一页接近"生产质量"

3. **用户参与感**
   - 用户全程看到进度和效果
   - 增强信心和满意度
   - 及时修正方向偏差

## 影响（Consequences）

### 积极影响

✅ **减少返工**：设计一次确认，编码过程中避免重复调整

✅ **提高效率**：Day 5 的 10+ 次迭代可以减少到 1-2 次

✅ **提升质量**：充分的设计思考会产生更好的最终结果

✅ **复用性强**：设计规范可被后续页面复用

✅ **用户满意**：提前规划设计，避免交付时的惊喜

✅ **团队协作**：设计规范清晰，新人可快速上手

### 潜在风险

⚠️ **前期时间增加**：tech-design 阶段需要多花 30 分钟进行设计确认

**缓解措施**：
- 这 30 分钟可以省去 implement-day 中的 2 小时调整时间
- ROI 高达 4:1（花 30 分钟前期确认 vs. 省 2 小时返工）

⚠️ **需要用户参与**：用户必须明确回答 4 个设计问题

**缓解措施**：
- 提供设计参考和示例
- AI 可以提出设计建议，用户只需选择
- 确认过程很快（10 分钟左右）

## 替代方案

### 方案 1: 设计系统优先（Design System First）

**优点**：
- 一次性建立完整设计系统
- 所有页面完全统一

**缺点**：
- 前期投入大（需要建立完整设计库）
- MVP 阶段不需要这么完整

**决策**：后续优化方向，MVP 不需要

### 方案 2: 迭代式设计（Progressive Design）

**优点**：
- 灵活，可以随时调整
- 不需要前期规划

**缺点**：
- Day 5 实证显示，迭代 10+ 次效率很低
- 设计不一致，各页面风格混乱

**决策**：不采用，因为效率太低

---

## 应用场景

### ✅ 适用

- 新功能的前端开发（有新页面需要设计）
- 需要统一视觉风格的功能
- Day 5+ 的所有前端开发

### ⚠️ 可调整

- 后端 API 页面：可能不需要复杂的设计确认
- 内部工具页面：可能可以简化设计规范

### ❌ 不适用

- 纯后端开发
- 没有前端界面的功能
