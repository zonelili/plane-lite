# ADR-001: Quality Assurance Workflow

**状态**: 已更新（增加测试层）
**日期**: 2026-03-24（初始）/ 2026-03-24（更新）
**决策者**: AI Agent + 用户
**相关文档**: `/docs/workflows/quality-assurance.md`, `/docs/lessons-learned.md`

---

## 背景（Context）

在 Plane Lite 项目的 Day 2 开发过程中，我们遇到了多个质量问题：

1. **版本兼容性问题**
   - Spring Boot 3.2 + MyBatis-Plus 3.5.5 不兼容
   - 这是一个已知问题，但在技术选型时没有验证
   - 反复犯同样的错误，浪费了大量时间

2. **代码与方案不一致**
   - Project 实体类定义了 `leadId` 字段
   - 但数据库初始化脚本 `V1__init.sql` 中缺少 `lead_id` 列
   - 导致运行时报错：`Unknown column 'lead_id' in 'field list'`

3. **环境配置问题**
   - MySQL 密码使用了默认值 'password'，未更新为实际密码
   - JWT secret 长度为 55 字符（440 bits），不满足 HS512 要求（>= 512 bits）
   - 这些问题在启动时才暴露，需要人工干预

### 根本原因分析

- **缺乏方案设计阶段的验证**：技术选型没有查阅官方兼容性文档
- **缺乏代码审查机制**：写完代码直接提交，没有对比 Plan 和实际代码
- **缺乏环境预检查**：启动失败才被动修复，而不是主动验证
- **AI Agent 记忆不持久**：流程没有文档化，AI 每次都需要重新思考

---

## 决策（Decision）

我们决定建立**四层质量保障机制**，将质量检查持久化到文档中：

### Layer 1: Plan Mode（方案设计）

**目标**：在开始编码前，验证技术栈兼容性和环境配置正确性

**实施方式**：
- 使用 `EnterPlanMode` 触发 Plan agent
- Plan 必须包含：
  - 技术栈兼容性验证（查阅官方文档）
  - 环境依赖清单（MySQL、Redis、JWT 配置要求）
  - 完整的 CREATE TABLE DDL（实体类字段与数据库列一一对应）
  - 验收标准（编译、启动、API 测试）

**触发条件**：
- 新功能开发（Day 2, Day 3 等）
- 技术选型变更
- 数据库表结构设计
- 架构调整

### Layer 2: Implementation（编码实现）

**目标**：在编码过程中自我检查，避免常见错误

**实施方式**：
- 每完成一个类后检查：
  - 实体类字段与数据库表列是否一致
  - DTO 是否有必要的校验注解
  - Service 方法是否有事务注解
  - Controller 是否有 Swagger 注解
- 每修改配置文件后检查：
  - 密码、密钥长度是否符合要求
  - 数据库连接 URL 是否正确
  - 端口配置是否冲突

### Layer 3: Testing（测试验证）

**目标**：独立验证功能是否正常工作，确保测试覆盖

**实施方式**：
- 触发时机：完成一个 Day 的开发任务后
- 测试步骤：
  1. **单元测试验证**：mvn test（后端）/ vue skill（前端）
  2. **API 功能测试**：手动测试（Postman/curl）
  3. **部署验证**：启动服务，健康检查
- 验收标准：
  - 所有单元测试通过
  - 测试覆盖率建议 >= 80%（不强制）
  - 所有 API 端点手动测试通过
  - 服务启动成功，健康检查通过

**触发条件**：
- 完成一个 Day 的开发任务（如 Day 2 所有模块）
- 修复重要 Bug 之后
- 准备进入 Code Review 之前

### Layer 4: Code Review（代码审查）

**目标**：验证代码是否符合设计，确保代码质量

**实施方式**：
- 使用 `superpowers:requesting-code-review` skill
- 检查内容：
  - 代码 vs Plan 一致性（API、字段、技术栈版本）
  - 实体类 vs 数据库表一致性（字段名转换、类型匹配）
  - 配置文件完整性（MySQL、JWT、Redis 配置验证）
  - 代码规范和最佳实践

**触发条件**：
- Layer 3: Testing 通过后
- 准备 Git commit 之前

## 理由（Rationale）

### 为什么选择四层机制？

**1. Plan Mode - 前置验证，避免方向错误**
- 技术选型错误的修复成本远高于前期验证成本
- 数据库表结构设计错误会导致大量返工
- 环境配置问题应该在 Plan 阶段就明确要求

**2. Implementation - 过程检查，及时发现问题**
- 实体类与数据库不一致是常见错误
- 编码过程中的自检成本最低
- 及时发现问题可以避免后续的级联错误

**3. Testing - 独立测试，确保功能正常**
- 测试是质量保障的核心环节，不应与代码审查混在一起
- 单元测试、API 测试、部署验证是不同的测试维度
- 测试失败应该立即返回修复，而不是等到审查时才发现
- 符合传统开发流程：开发 → 测试 → 审查 → 提交

**4. Code Review - 最后防线，确保代码质量**
- 在测试通过的基础上，审查代码是否符合设计
- 验证代码 vs Plan 一致性、实体 vs 数据库一致性
- 检查代码规范和最佳实践
- 人工审查容易遗漏细节，自动化审查更可靠

### 为什么要增加独立的测试层？

**问题背景**：
在 Day 2 开发完成后，用户指出当前的三层机制缺少独立的测试环节。在传统开发团队中，流程应该是：
```
开发完成 → 测试团队测试 → 代码审查 → 提交
```

但我们的三层机制将测试和审查混在一起（Layer 3: Code Review 中包含编译、启动、API 测试），导致：
- **职责不清**：测试验证（功能是否正常）和代码审查（代码是否符合设计）混在一起
- **错误发现延迟**：功能 Bug 应该在测试阶段发现，而不是在审查阶段
- **流程不符合直觉**：违反了"测试团队独立验证"的传统流程

**改进方案**：
- 将测试验证独立为 Layer 3: Testing
- 将代码审查改为 Layer 4: Code Review
- Testing 重点验证功能是否正常（单元测试、API 测试、部署验证）
- Code Review 重点验证代码是否符合设计（代码 vs Plan、实体 vs 数据库、代码规范）

### 为什么要文档化？

**问题**：AI Agent 的记忆不持久
- 每次会话都需要重新思考流程
- 容易遗漏关键检查项
- 无法积累经验

**解决方案**：将流程文档化
- 创建 `/docs/workflows/quality-assurance.md`
- 在 `CLAUDE.md` 中引用
- 定义明确的触发条件和检查清单

**好处**：
- AI Agent 可以记住流程并自动执行
- 新的 AI Agent 可以快速学习流程
- 流程可以持续优化和改进

### 为什么不使用 Git hooks？

**考虑过的方案**：pre-commit hook 自动检查

**选择不用的原因**：
- Git hooks 需要本地环境配置，增加了开发门槛
- AI Agent 驱动的开发需要更灵活的触发机制
- 文档化的流程更容易被 AI 理解和执行
- 后续可以逐步引入 Git hooks，但不作为第一步

## 影响（Consequences）

### 积极影响

✅ **减少返工**：技术选型错误、实体类与数据库不一致等问题将显著减少

✅ **提高代码质量**：自动化审查可以发现人工容易遗漏的问题

✅ **职责分离清晰**：测试验证功能，审查验证设计，各司其职

✅ **符合传统流程**：开发 → 测试 → 审查 → 提交，符合开发者心智模型

✅ **加速开发**：虽然增加了检查步骤，但减少了修复 Bug 的时间

✅ **知识积累**：流程文档化后，经验可以被持续积累和改进

✅ **AI Agent 友好**：明确的触发条件和检查清单，AI Agent 可以自动执行

### 潜在风险

⚠️ **增加流程复杂度**：四层机制可能让开发流程变得复杂

**缓解措施**：
- 简单修改（如 Bug 修复）可以跳过 Plan Mode
- 自动化测试和审查可以在后台执行，不阻塞开发
- 提供清晰的触发条件说明，避免混淆

⚠️ **依赖文档维护**：文档过时会导致流程失效

**缓解措施**：
- 每次流程改进后立即更新文档
- 在 `lessons-learned.md` 中记录问题和改进
- 定期回顾和优化流程

⚠️ **测试能力有限**：部分测试能力尚未完善

**缓解措施**：
- MVP 阶段使用手动测试（Postman/curl）
- 后续逐步引入 API 测试框架（RestAssured）
- 规划 E2E 测试工具（Playwright）

## 替代方案（Alternatives Considered）

### 方案 1: 仅依赖人工审查

**优点**：
- 简单，无需额外工具
- 灵活，可以根据具体情况调整

**缺点**：
- 人工审查容易遗漏细节
- 无法保证一致性
- AI Agent 难以执行

**决策**：不采用，因为无法保证质量

### 方案 2: 引入 CI/CD 自动化测试

**优点**：
- 完全自动化
- 可以集成到 Git 工作流

**缺点**：
- 需要配置 CI/CD 环境
- AI Agent 驱动的开发不一定依赖 Git
- 过于重量级，不适合 MVP 阶段

**决策**：不作为第一步，但可以作为后续优化方向

### 方案 3: 使用 Linter 和静态分析工具

**优点**：
- 可以自动检查代码风格和常见错误
- 工具成熟，易于集成

**缺点**：
- 无法检查业务逻辑一致性（如实体类与数据库不一致）
- 无法验证技术栈兼容性
- 无法检查环境配置

**决策**：作为补充，但不能替代三层机制
